## 二、逐项质询

### 质询 1（证据充分性）：N-01 修复"彻底"判定言过其实 —— event_time 提取路径在 handler 步骤中缺失

**审查报告声称**（§二 N-01）：修复彻底，"覆盖了审查报告和质询报告的全部关切"。

**质询依据**：

设计文档 §4.2 幂等性保障（行 755-760）在父级章节中写明了原则：

> "写入 `timestamp` 字段时优先使用 IoTDA 推送的 `event_time`（毫秒级精度...）"

但问题出在子章节的具体 handler 步骤中：

- **§4.2.1 处理逻辑**（行 807-808）：
  - 步骤 1："解析 `notify_data.body.services[0].properties` 中的各字段"
  - 步骤 2："写入 `sensor_snapshot` 表（使用 `INSERT ... ON CONFLICT (device_id, timestamp) DO NOTHING`）"
  - `event_time` 位于请求体顶层 `notify_data` 的同级字段（行 776 `"event_time": "2026-06-30T10:15:30Z"`），**不在 `services[0].properties` 内**。步骤 1 的解析范围不涵盖 `event_time`。步骤 2 使用 `timestamp` 但未说明从何处取值。

- **§4.2.2 处理逻辑**（行 857-858）：
  - 步骤 1："写入 `disease_records` 表（使用 `INSERT ... ON CONFLICT (device_id, timestamp, disease_type) DO NOTHING`）"
  - 该 handler **根本没有"解析"步骤**（与 §4.2.1 不同），更未提及从何处获取 `timestamp` 的值。

**影响评估**：实现者若只阅读 handler 步骤（最常见的开发模式——按接口逐节查看），将无法从步骤描述中获知 `timestamp` 应使用 `event_time`。可能直接使用 `CURRENT_TIMESTAMP` 或服务端时间，导致 IoTDA 重试推送时 `INSERT` 产生不同的 `timestamp` 值，ON CONFLICT 无法命中，重复数据入库。幂等性保障的核心前提（timestamp 取值正确）在 handler 步骤层面**未操作化**。

**审查报告自身评价**（行 180）：
> "§4.2.1/§4.2.2 处理逻辑中未逐步骤重复声明'使用 event_time' —— 可接受——统一说明已明确适用范围"

**质询判定**：此评价过于宽容。父级章节的原则性声明与子章节的操作步骤之间存在 **实现断层**。合理的做法应是在 handler 步骤中至少加一句"timestamp 取值 = notify_data.event_time"，使操作性完整。此项不构成 N-01 修复失败，但审查报告将其定性为"可接受"而完全不予提及，是证据充分性上的不足。

**质询结论**：N-01 修复在**设计意图层面**正确但在**操作步骤层面**不完整。审查报告的"彻底"判定高估了修复的完备性。建议补充为 C-01（建议级新增发现）。

### 质询 2（覆盖完备性）：审查遗漏 —— disease_records API 示例中 action_taken 与决策矩阵矛盾

**审查报告声称**（§四）："未发现因本轮修改引入的新设计缺陷。"

**质询依据**：

§4.4.1 病虫害记录 API 成功响应示例（行 1133-1138）：

```json
{
  "severity": "Moderate",
  "severity_code": 2,
  "action_taken": "auto_spray"
}
```

但决策规则矩阵 §2.4（行 332）明确：

| # | 病虫害 | severity_code | 条件满足—自动动作 |
|---|--------|:---:|:----------------:|
| 2 | rust | 2 | **无自动** |

severity_code = 2 的决策规则是"**无自动**"动作。自动喷淋 (`spray ON`) 仅在 severity_code = 3 时触发（行 333："任意 → `spray ON`"）。

示例数据 `"action_taken": "auto_spray"` 与决策矩阵 `"无自动"` **直接矛盾**。实现者参照此示例可能误判：以为 severity 2 也会触发自动喷淋，从而在实现决策引擎时对所有 severity >= 2 的检测都写入 `action_taken = "auto_spray"`。

**影响评估**：这是一个文档示例级缺陷，不涉及设计逻辑本身的错误（决策矩阵是正确的），但作为 API 规范中的示例数据，其误导风险不可忽略。

**质询结论**：审查报告在"新缺陷检查"巡查中遗漏了此矛盾。建议补充为 C-02（建议级新增发现）。

## 三、S-05 与 S-06 升级评估

### S-05（命令列表中区分 IoTDA vs 本地控制路径）

**审查报告评级**：低 / 建议 / 不阻塞

**质询分析**：

设计文档中实际上已**隐含**做出了区分：
- `POST /api/v1/command` 的 8 条命令（行 1284-1293）**全部经由 IoTDA 下发**（§4.5.1 步骤 4 明确"调用华为云 IoTDA 命令下发 API"）
- "纯本地控制动作"（§4.5.1 行 1260）指的是**不经过 `POST /api/v1/command` 接口**的内部触发路径（告警引擎直接调用 GPIO 控制、定时自检等）
- 二者的执行路径在架构上已经是分离的，只是命令列表中的 8 条字符串在两种路径下**可能语义重叠**（如 `beep ON` 既可通过 API 下发，也可被本地告警引擎触发）

结论：区分已经存在，仅在命令列表中缺少路径标注。**维持建议级，不升级。**

### S-06（sensor_daily_aggregation UNIQUE 约束与索引冗余）

**审查报告评级**：低 / 建议 / 不阻塞

**质询分析**：

DDL（行 512-515）：
```sql
UNIQUE (device_id, agg_date);                                          -- 行 512
CREATE INDEX idx_agg_device_date ON sensor_daily_aggregation (device_id, agg_date);  -- 行 515
```

PostgreSQL/KingbaseES 的 UNIQUE 约束自动创建唯一 B-tree 索引。同列集的普通 B-tree 索引确实完全冗余——唯一索引已覆盖等值查询和范围扫描（以 `device_id` 为前缀）。冗余索引额外占用磁盘空间并在每次 INSERT/UPDATE 时多维护一棵 B-tree，但：
- 日聚合表每天每设备仅写入 1 行，写入开销可忽略
- 查询性能不受影响（优化器选择唯一索引即可）

结论：纯性能优化问题，不影响功能正确性。**维持建议级，不升级。**
