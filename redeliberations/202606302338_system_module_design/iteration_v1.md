# 再审议判定报告（v1）

## 判定结果

RETRY

## 判定理由

### 一、严重问题确认

诊断报告发现 **C-03**（`system_architecture_relationship.md` 文件不存在）为严重问题，质询报告对此未提出异议。该问题导致设计文档 §1.1 术语等价说明、§2.5 表名对照、§5.1 目录结构、§6.6 命名协调等多处关键引用悬空，架构定义的上下文完整性受损。

### 二、高优先级问题核实

经质询报告对严重度调整后（C-01/C-02 由"严重"降为"高"），残余 6 项高优先级问题：

1. **C-01**: `control_logs` 表缺少 `command_id` 字段，命令应答匹配链路断裂。设计文档虽已标注为已知缺陷，但当前 Schema 与业务流程不匹配，属功能阻塞。
2. **C-02**: HTTP API 无认证/授权机制，在公网 VPS 部署场景下存在安全风险，尤其 IP 白名单方案不适用移动端（鸿蒙 App）。
3. **H-01**: 规格说明书（§2.3 华为云）与模块设计（VPS KingbaseES）在数据存储位置上存在矛盾，设计文档 §6.1 虽给出 VPS 部署理由但未显式对比规格书原文并给出分歧仲裁说明。
4. **H-02**: Python 上位机"历史数据导出（CSV/Excel）"功能在 §2.7 有职责定义但 §4.8 接口清单中无对应 API 端点，属功能-接口不匹配。
5. **H-04**: `GET /api/v1/sensor/latest` 多设备场景下响应格式（单对象）与接口语义（"返回所有设备各自最新一条"）存在直接矛盾，客户端无法解析。
6. **H-06**: 光照报警阈值 `light < 100`（量程 0-100）导致几乎持续误报，属跨文档一致的设计错误。

### 三、中优先级问题汇总

经质询调整严重度后，仍有 11 项中等问题（H-03/H-05 由高降为中，M-02/M-04 由中降为低，S-01/S-02 为质询补充），涵盖：
- 数据结构不完备（缺少独立 `images` 表、设备信息表）
- 边界条件行为未定义（Webhook 幂等性、决策规则矩阵条件不满足时行为）
- 部署方案不完整（Redis/dict 方案选择未定、API 健康检查缺失）
- 设备端 `service_id` 的 MQTT 携带方式未明确

### 四、综合判断

依据判定标准，审查报告包含严重和一般等级问题（C-03 确认为严重，6 项高/11 项中为一般等级），**不满足 PASS 条件**。根据 judge.md §判定标准 §RETRY："审查报告包含严重或一般等级的问题"，裁定 **RETRY**。

当前设计存在以下需在详细设计前修复的阻塞性问题：
- 数据库 Schema 需变更（4 张新表/字段、3 项索引调整）
- API 端点需增加/变更（至少 4 个新增端点、4 个现有端点变更）
- 跨文档协同修订（规格说明书、DATA_INVENTORY、设计文档三向对齐）
- 安全性补全（API 认证、MQTT TLS）

建议修订方：参考本判定报告及质询报告（`b_v1_challenge_v1.md`）的严重度调整建议和补充问题清单，对组件 A 产出进行系统性修订后重新提交审查。
