**质询意见**: **Agree（认可）**，但严重度建议从"严重"降为"高"。

### C-02: 系统对外暴露的 HTTP API 无任何认证/授权机制

**质询意见**: **Agree（认可）**，但严重度建议从"严重"降为"高"。

**理由**：
- §4.1 明确声明"初版可暂用无认证"是一种**有意为之的设计权衡**而非设计遗漏。审查报告将之列为"严重"级别，使用了"任何知道 IP 和端口的攻击者均可直接调用设备控制接口"等风险语言——技术上正确，但未对设计文档已做出的权衡给予权重。

- 但综合来看，这是对"初版"设计决策的质量挑战，而非对遗漏的发现。建议在修订中保留问题但标注为"对既有设计决策的质疑（Design Decision Challenge）"。
- **证据来源**: `a_v1_imported.md` §4.1 "认证方式"行。

**修订建议**: 严重度降为"高"，并在问题描述中注明该设计决策已在文档中标记为有意简化，审查是从安全最佳实践角度提出升级建议。

### H-03: 图片上传接口在无关联 `disease_record_id` 时元数据无法持久化

**质询意见**: **Agree（认可）**，但严重度建议从"高"降为"中"。

**理由**：
- 问题的技术分析（图片无独立表、孤立文件风险）是正确的。
- 但实用意义上，此问题的严重度被高估：(a) 图片上传的主要场景是关联已有识别记录（用户在现场看到病虫害后拍照补充图片证据），此时 `disease_record_id` 几乎总是被提供；(b) 不提供 `disease_record_id` 的上传场景（如纯粹的文件上传）在当前的业务流中并非核心路径。

- **证据来源**: `a_v1_imported.md` §4.7.1。

**修订建议**: 降为"中 (Medium)"，或保留"高"但在描述中明确区分"缺少独立 images 表的结构性缺陷"与"不传 disease_record_id 的边界条件"两个层面。

### H-04: `GET /api/v1/sensor/latest` 多设备场景响应格式与单设备场景冲突

### H-05: 湿度报警阈值与 DATA_INVENTORY 中传感器量程存在边界矛盾

**质询意见**: **Agree（认可）**，但存在过度解读，建议降为"中"。

**理由**：
- 审查报告称"若传感器量程上限即为 90%，则 `humidity > 90.0%` 的条件在量程范围内**永远无法触发**（最大只能到 90.0，不大于 90.0）"。这一推理在数学上是正确的，但忽略了几个因素：
  - DHT11 数据手册定义的量程 20-90% RH 是**保证精度范围**（±5%），并非硬上限。在接近冷凝的高湿环境中，传感器读数可能超出 90% (如 92-95%)。
  - DATA_INVENTORY.md §2.2 将"范围"写为"20 ~ 90%"，但未明确这是"量程范围"还是"正常工作范围"。若为后者，则超出 90% 的读数仍可能出现（精度下降）。
- 审查报告的实用建议（调整为 `>= 90.0%` 或 `>= 85.0%` 留余量）是合理的工程实践，但"永远无法触发"的断语不够严谨。
- **证据来源**: `DATA_INVENTORY.md` §2.2；DHT11 datasheet (not in repo)；`a_v1_imported.md` §2.1。

**修订建议**: 降为"中"，问题描述中修改"永远无法触发"为"在传感器额定精度范围内难以可靠触发"。

### H-06: 光照报警阈值导致几乎持续误报

**质询意见**: **Agree（认可）**，严重度维持"高"。

**理由**：
- `light < 100`（其中 light 范围为 0-100）实际上等价于"光照未达到满量程时持续报警"。这是一个设计错误。
- 该阈值同时存在于 DATA_INVENTORY.md §2.3 和设计文档 §2.1，属跨文档一致的错误，需同步修正。
- 审查报告的建议（调整为 `light < 20` 或 `light < 10`）合理。但需补充：该阈值在嵌入式端（板端 alarm_flag 计算）和 Python API 端（决策引擎阈值）应一致修改，审查报告建议的修订范围仅提到了 DATA_INVENTORY.md，未明确要求同步修改嵌入式端的 alarm_flag 触发条件。
- **证据来源**: `DATA_INVENTORY.md` §2.3；`a_v1_imported.md` §2.1。

**质询意见**: **Disagree（反驳）**，建议移除。

**理由**：
- 审查报告批评 PyQtGraph 的 `>60fps` 渲染能力对 10s 刷新频率的场景不必要，但混淆了**数据刷新频率**与**图形渲染能力**两个不同概念。
- 在 GUI 图表应用中，即使数据仅每 10s 更新一次，用户仍可能进行以下需要高渲染帧率的操作：平移/缩放图表、窗口调整大小、多面板切换动画、鼠标悬停 tooltip 交互。PyQtGraph 的 `>60fps` 声明指的是**渲染管线的刷新能力**而非数据轮询频率。
- 审查报告的建议（改为"在数据点累积较多时仍能保持流畅渲染"）确实比原文更准确地表达了选择理由，但这更恰当定性为**措辞改进建议**而非**设计缺陷**。
- **证据来源**: `a_v1_imported.md` §2.7 图表库选择理由。

## 二、遗漏的重要问题（补充提出）

### S-01 (建议严重度: 中): 设计文档未定义设备端 `service_id` 在 MQTT 消息中的携带方式

- **位置**: `a_v1_imported.md` §2.1、§2.3、§3.1
- **问题描述**:
  IoTDA Webhook 路由规则（§4.2 三 Webhook 路由规则配置方案）依赖 `service_id = "farmeye_env"` 和 `service_id = "farmeye_ai"` 区分传感器数据与 AI 识别结果的转发目标。然而在设备端 MQTT 上报部分（§2.1 上报数据字段表、§3.2 流 A 序列图、DATA_INVENTORY §3.1 属性上报 payload），`service_id` 仅出现在 IoTDA 转发后的 JSON 结构中，**设备端如何将不同种类的数据分别打包为不同 service_id 的 MQTT 消息未被明确描述**。嵌入式端开发者无法从当前设计文档中得知"应在何种条件下、以何种 service_id 上报"。
- **建议**: 在 §2.1 中增加"数据上报策略"小节，明确：(a) 传感器数据（10s 周期）使用 `service_id = "farmeye_env"`；(b) AI 识别结果（事件触发）使用 `service_id = "farmeye_ai"`；(c) 给出对应的 MQTT payload JSON 模板。
- **证据来源**: `a_v1_imported.md` §4.2 路由规则表 vs §2.1 上报数据字段表 vs §3.2 流 A/B。

### S-02 (建议严重度: 中): 设备在线判定依赖的 Redis 未在部署方案中体现

- **位置**: `a_v1_imported.md` §2.4（设备在线/离线判定机制）；§5.2（docker-compose.yml）
- **问题描述**:
  §2.4 明确写道"设备在线状态维护在内存缓存（Redis 或进程内 dict）中"。若选择 Redis，则 docker-compose.yml（§5.2）中未包含 Redis 服务。若选择进程内 dict，则文档未分析多 worker 进程场景下（如 Uvicorn `--workers 4`）dict 方案的不一致性问题——不同 worker 的 dict 各自独立，设备在线状态将出现不一致。
- **建议**: (a) 明确选择一种方案（建议进程内 dict + 单 worker 部署以降低复杂度，适合课程项目）；(b) 若选 Redis，在 docker-compose.yml 中补充 Redis 服务。
- **证据来源**: `a_v1_imported.md` §2.4 第三段 vs §5.2。

### S-03 (建议严重度: 低): API 服务 docker-compose 缺少 healthcheck

- **位置**: `a_v1_imported.md` §5.2 docker-compose.yml
- **问题描述**:

- **建议**: 在 §5.2 的 api 服务定义中增加 healthcheck 配置，引用 `/api/v1/health` 端点（与 M-05 建议协同）。
- **证据来源**: `a_v1_imported.md` §5.2。

### S-04 (建议严重度: 低): `GET /api/v1/command/logs` 的 filter 能力不完整

- **位置**: `a_v1_imported.md` §4.5.2
- **问题描述**:
