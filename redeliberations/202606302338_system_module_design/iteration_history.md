### 5. 本轮关键经验教训

1. **跨文档一致性验证是阻塞性检查项**：规格说明书、数据清单、架构关系文档与模块设计文档之间的不一致（存储位置、传感器清单、术语命名）会导致设计意图冲突，应在模块设计阶段开始前完成基线对齐。
2. **引用文件的完整性应在审查前自检**：设计文档多处依赖不存在的 `system_architecture_relationship.md` 提供关键术语对照，该文件应在产出设计文档的同时创建，而非延期处理。
3. **API 边界条件设计不可省略**：多设备响应格式、无关联记录时的元数据处理、分页上限行为等边界条件虽非主线流程，但缺失时会导致客户端实现时产生歧义和运行时错误。
4. **安全设计不应被完全推迟**：即使在课程项目/初版中，暴露在公网的 API（尤其是设备控制接口）至少应具备最小化认证（如 API Key），"暂不实现"应在风险评估下做出而非仅凭简化考量。
5. **已知缺陷应显式标注但不可豁免**：设计文档对 `command_id` 缺失的"详细设计阶段修复"标注是良好实践，但这类延期项应被显式追踪（如 TODO 列表），确保在进入详细设计时不遗漏。
6. **严重度标定应区分"遗漏"与"有意简化"**：质询报告揭示审查将 C-01/C-02 标注为"严重"而未区分设计文档的自我认知（已知缺陷 vs 完全未发现），导致修订优先级失真——这一教训对后续审查的严重度判定标准具有方法论价值。
7. **传感器量程与报警阈值的匹配应基于数据手册而非惯性**：H-05（湿度阈值）和 H-06（光照阈值）暴露了传感器实际特性（精度范围、量程边界）与软件阈值之间的脱节，应在传感器选型阶段建立"传感器量程-报警阈值"映射表。

### 6. 本轮关键经验教训

1. **DDL 与应用逻辑的一致性验证必须精细到 SQL 语法层面**：N-01 暴露了设计审查停留在"语义正确性"层面的风险——DDL 中的 `CREATE INDEX` 与处理逻辑中的 `ON CONFLICT` 在语义上看起来配套，但数据库语法层面存在致命不兼容。后续审查应将 SQL 可执行性验证纳入标准检查项。

2. **API 响应规范与 DDL 的双向对齐检查不可省略**：N-02（devices 表字段缺失）、O-01（disease_records 响应遗漏联动字段）、`/sensor/history` 字段不一致，三者均属于同一类设计错误——DDL 定义了字段但 API 响应遗漏，或 API 响应包含了 DDL 中不存在的字段。建议在后续设计审查中建立"DDL-API 响应矩阵"逐字段对照检查。

3. **数据保留策略的查询端闭环不可忽视**：O-02 揭示了典型的设计断层——数据定义了完整的"采集→聚合→保留"链路，但缺少"查询"环节，导致聚合数据成为无法访问的"暗数据"。数据全生命周期设计应始终以"客户端如何获取"为终点。

4. **修订说明的自我声明需要独立验证**：本轮审查通过逐项验证修订说明中的 30 项修复声明，确认了"声明修复但未实际修改"的虚假声明不存在。这一做法对保障修订质量具有方法论意义——修订说明应作为审查的检查清单，而非信任凭证。

5. **质询报告的覆盖广度补偿价值**：首轮审查 29 项 + 本轮审查 2 项 + 残留 3 项，诊断报告看似覆盖完整；但质询报告补充发现了 2 项中等和 2 项低等问题（O-01~O-04），说明单一审查视角存在固有盲区，质询机制有效发挥了覆盖广度补偿作用。
