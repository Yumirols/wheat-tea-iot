## 质询项

### CH1. H7 严重度分类与需求范围存在逻辑矛盾（逻辑完整性：通过）

**位置**：`b_v1_diag_v3.md` H7（第 167–192 行）

**问题描述**：

审查报告将 H7 `PollingManager` 接口未预留 WebSocket 升级扩展点 判定为**高严重度**（"阻塞实现"）。然而需求 §2.2 明确表述：

> *"v1.0 采用 HTTP 轮询 10s 间隔，**后续可升级**为 WebSocket"*

需求文本明确将 WebSocket 定位为**后续升级路径**，而非 v1.0 的约束条件。当前 `PollingManager` 的 `start(key, fn, interval)` 接口完全满足 v1.0 HTTP 轮询的所有要求。

**矛盾分析**：
- 审查报告自身对 HIGH 严重度的定义隐含"阻塞实现"（即不修复则无法编码实施）
- 但 WebSocket 扩展的缺失**并不阻塞** v1.0 的 HTTP 轮询实现
- 将"后续可升级"的需求扩展方向归类为当前版本的高严重度问题，与需求文档的明确范围相悖

**补充依据**：参考项目 `reference/zhihui` 同样使用直接 `setInterval` + HTTP 轮询（DisplayPage.ets:270-273），未引入轮询调度器抽象层，证明 `PollingManager` 的 v1.0 接口已是相对于参考项的架构提升。

**影响判定**：严重度分类误差导致报告整体严重度统计数据失真（10 HIGH → 实质 9 HIGH），影响读者对设计成熟度的判断。

### CH2. H1 对 PixelMap 解码路径的处方未充分评估更简替代方案（证据充分性）

**位置**：`b_v1_diag_v3.md` H1（第 27–44 行）

**问题描述**：

审查报告断言 `ArrayBuffer` → `PixelMap` 解码是 ImageViewer 展示的唯一可行路径，并据此要求 `ImageService` 暴露 `getImagePixelMap()` 方法。但设计中存在另一条更简洁的路径未被评估：

设计 §15 明确描述 ImageViewer "使用 `Image` 组件的网络源能力"，即 `<Image src={url}>` 直接加载。若 API 返回的 `image_path` 字段为可访问 URL（或可通过 `baseURL + image_path` 拼接为完整 URL），则无需经过 `ImageService.getImage()` 和 `ArrayBuffer` 路径。

**证据缺口**：
- 审查报告未分析 `DiseaseRecord.image_path` 字段的语义（URL 还是 image_id）
- 未评估"直接 URL 路径"与"二进制 + PixelMap 解码路径"在 v1.0 场景下的取舍
- 若 `image_path` 为 URL，则 `ImageService.getImage()` 的二进制路径可推迟到后续版本，H1 和 H4 的紧急度均显著降低

**补充**：`GET /image/{image_id}` 接口的存在不意味着 `image_path` 存储的是 image_id。实际 API 设计中 `image_path` 可以是完整 URL（由后端在疾病记录创建时拼接生成），前端仅消费。

**影响判定**：未充分评估替代路径即给出高严重度判定，证据链条不完整。

### CH3. M4 对设计文档的必要内容边界判断偏严（覆盖完备性）

**位置**：`b_v1_diag_v3.md` M4（第 279–291 行）

**问题描述**：

审查报告将"`hilog` 日志策略未在设计中体现"列为中严重度问题，认为"未满足需求约束"。但需求 §3 技术栈中"日志：`hilog`"属于**技术选型说明**，而非要求设计文档必须定义日志策略的行为约束。OOD 设计文档的职责是架构、组件划分、协作关系和关键行为契约——日志调用方式属于编码阶段的实现惯例，可在编码规范中约定而非设计文档中展开。

**影响判定**：对设计文档的内容边界期望过高，可能导致设计文档膨胀至实现规范级别。
