## 质询 1：M1 修复验证与 S5 持续存在项之间存在逻辑矛盾（逻辑完整性）

**位置**：M1 验证表（第44行）与 S5 持续存在项（第165行）

**问题描述**：

审查报告在 M1 验证表中判定：
> 定义 `PollingCallback` 类型 → ✅ **完整修复**

但在 S5 持续存在项中记录：
> `PollingCallback` 类型未显式导出为独立类型定义 → ⚠️ 持续存在

这两条陈述构成**直接逻辑矛盾**：

- 若 M1 的修复要求"定义 `PollingCallback` 类型签名"已被 **完整修复**，则 `PollingCallback` 应在设计文档的核心抽象或 `common/models.ets` 中以显式类型定义存在。若如此，S5 的"未显式导出"即为错误陈述。
- 若 S5 成立（类型确实未显式导出为独立类型定义），则 M1 的修复并未真正完成——设计文档仅在修订说明中提及签名字符串，核心抽象 §9 `PollingManager` 中仅用 `fn` 参数名称隐式描述回调行为，未在任何位置给出 `type PollingCallback = () => Promise<void>` 或等效的显式类型定义。

审查报告不能对**同一个质量维度**（PollingCallback 的类型定义完整性）在同一份报告中同时给出 ✅完整修复 和 ⚠️持续存在 的冲突判定。二者必有一误。

**严重程度**：中 — 影响审查报告结论的内部一致性，削弱可落地性综合评价的可信度

**质询要求**：澄清 M1 的"完整修复"判定标准与 S5 的"持续存在"判定标准之间的逻辑关系，或修正其中一方的判定。

## 质询 2：N2 `connectivityStatus` 改进建议存在双数据源隐患（逻辑完整性）

**位置**：N2 改进建议（第107–110行）

**问题描述**：

审查报告的 N2 发现指出每个页面独立的 `@State connectivityStatus` 缺乏状态转换闭环。随后提出的改进建议为：

> 在 `HttpClient` 或 `api.ets` 层增加模块级 `connectivityStatus` 信号（如模块级 `let globalConnectivity: 'loading' | 'online' | 'offline' = 'loading'`），作为统一的连接状态源

此建议存在以下未解决的矛盾：

- 设计文档中每个页面已声明独立的 `@State connectivityStatus`（§12 页面组件）
- 建议新增**模块级** `globalConnectivity` 作为"统一状态源"
- 若页面使用模块级信号替换 `@State`，则需解释 ArkUI `@State` 如何绑定模块级 `let` 变量——ArkTS 中 `@State` 只能装饰 struct 的成员变量，无法直接绑定到模块级 `let`
- 若同时保留 `@State` 和模块级信号，则形成双数据源，谁为权威、如何同步均未定义

改进建议引入了一个新的未解决的架构问题，但审查报告未对此进行评估，也未给出具体的数据流同步方案。

**严重程度**：低-中 — 建议本身存在实现不确定性

**质询要求**：补充双源合一的具体方案（如：模块级信号仅用作统一判定源，页面通过每次 Service 调用后同步至 `@State` 的完整转换矩阵），或取消模块级方案推荐、改用其他机制。

## 质询 3：N1 改进建议 C 引用可能不存在的 ArkUI API（证据充分性）

**位置**：N1 改进建议方案 C（第87行）

**问题描述**：

方案 C 建议：

> 成功时等待父组件 `@Prop` 更新 `isOn` 后在 `aboutToUpdate` 中同步 `displayOn`

`aboutToUpdate` 并非 ArkUI API 21 的标准组件生命周期回调。ArkUI 组件的标准生命周期包括 `aboutToAppear` 和 `aboutToDisappear`。**`aboutToUpdate` 不存在于 ArkUI 的公开 API 中**。

若此 API 不存在，方案 C 不可落地，不应作为推荐方案列入改进建议（即使标注为"v1.0 最小改动"）。审查报告的改进建议应在其所依据的技术栈约束内可行。

**严重程度**：低 — 不影响核心诊断，但削弱改进建议的可执行性

**质询要求**：确认 `aboutToUpdate` 在 API 21 中的可用性，或移除方案 C、或替换为基于 `@Link` 的方案 A 并标注为唯一推荐。

## 质询 4：N4 定时漂移改进建议存在策略逻辑未闭合（逻辑完整性）

**位置**：N4 改进建议（第132行）

**问题描述**：

审查报告指出串行模式的定时漂移行为预期未定义，但改进建议本身也以问句形式留下了**未闭合的决策**：

> 需明确策略：超时后是立即执行下一个、还是等待完整间隔？

作为诊断报告，应当在提出问题的同时给出**确定的推荐策略**，而非将决策问题抛回给实现者。否则该发现与未发现无异——实现者同样面临"如何决策"的困惑。

**严重程度**：低 — 建议半成品，未完成分析闭环

**质询要求**：在审查报告中给出明确的推荐策略（如推荐"等待完整间隔"或其他方案），并附上理由。
