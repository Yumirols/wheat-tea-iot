## 判定理由

组件B第2轮诊断报告（`b_v2_diag_v2.md`）识别出 **2 项中严重度（一般等级）问题** 和 **4 项低严重度问题**：

## 需要解决的问题（仅 RETRY 时存在）

- **问题描述**：N1 — `ControlButton` 使用 `@Prop` 接收 `isOn`，但 `@Prop` 只读导致乐观 UI 翻转无法写回父组件状态，回滚时存在状态不一致风险
- **所在位置**：核心抽象 §16 `ControlButton`（第307–318行）
- **严重程度**：一般
- **改进建议**：采用方案 A — `ControlButton` 改用 `@Link` 接收 `isOn`，乐观 UI 通过 `this.isOn = targetState` 直接修改父组件状态；或备选方案 B：维持 `@Prop` + 内部 `@State displayOn` 覆盖显示。需在核心抽象 §16 中明确选择并补充 `@Link` 决策说明

- **问题描述**：N2 — 各页面 `@State connectivityStatus` 的完整状态转换闭环未定义：触发时机（何时 `loading→online`）、离线判定（谁负责切换为 `offline`）、恢复机制（`offline→online` 条件）、跨页面一致性均未明确
- **所在位置**：核心抽象 §12 页面组件（第256行）、错误处理策略弱网韧性表（第510行）
- **严重程度**：一般
- **改进建议**：采用诊断报告提供的纯页面级状态转换矩阵，在 `loadData()` 的统一 catch 中根据每次 Service 调用结果维护 `connectivityStatus`，并在错误处理策略中补充转换矩阵表
