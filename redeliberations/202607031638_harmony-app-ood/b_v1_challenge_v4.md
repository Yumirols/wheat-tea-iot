## CH1 — 覆盖完备性：IndexPage 传感器数据实时刷新机制未评估

**位置**：`b_v1_diag_v4.md` 全文

**问题**：

需求 §2.1 明确要求**"环境参数实时展示卡片：温湿度、光照、CO2、土壤 NPK、信号强度等传感器数据的实时数值显示"**。但 `b_v1_diag_v4.md` 在审查 OOD 设计方案 `a_v1_design_v3.md` 时，未评估 **IndexPage 首页是否具备传感器数据的实时刷新能力**。

设计文档场景 A 显示 IndexPage 的轮询调度为：
```
PollingManager.start('index_alarm', 10000)
   └─ 每 10s: AdvisoryService.getAdvisory() → 检测新告警
```

IndexPage 上的 `SensorCard` 群组仅在 `aboutToAppear` 时通过 `SensorService.getLatest(deviceId)` 获取一次静态快照，**没有设置传感器数据轮询**。这意味着用户停留在首页时，温湿度等环境参数是**静态快照而非实时数值**，与需求 §2.1 的"实时展示"矛盾。

DashboardPage 虽有 `dashboard_sensor` 轮询（10s 刷新传感器数据），但用户通常将首页（IndexPage）作为默认停留页面。如果首页传感器卡片不刷新，核心体验在默认场景下是"准实时"而非"实时"。

**判定**：审查报告未覆盖此可落地性缺口，属于审查维度的覆盖遗漏。

## CH2 — 证据充分性：H2（ChartView Canvas 复杂度）工时估算缺乏实证支撑

**位置**：`b_v1_diag_v4.md` 第 67 行

**问题**：

H2 判定 ChartView 在 ArkUI API 21 中缺少原生图表组件，给出了"**开发量预估 5–10 人日**"的量化结论，但：

1. **无出处**：未引用任何 ArkUI Canvas `CanvasRenderingContext2D` API 性能基准数据、官方文档限制或社区实践
2. **无分解**：未将 5–10 人日的工作量分解为坐标轴系统、数据映射、触摸交互等子任务的独立估算
3. **无对照**：未参考 `reference/zhihui` 中是否有可复用的渲染模式或可借鉴的 Canvas 实现经验
4. **无依据说明数据点量级**：传感器历史数据点数量级未知（50 点 / 500 点 / 5000 点？），不同量级下 Canvas 绘制复杂度差异巨大，该估算缺乏标定前提

作为"可落地性"审查报告，一个关键的可落地性判断（是否存在框架缺口导致开发周期不可控）应当给出可追溯的估算依据。

**判定**：H2 的高严重度定性和工时量化缺乏证据链支撑。

## CH3 — 逻辑完整性：H1 与 H4 之间的依赖关系未处理，结论存在潜在矛盾

**位置**：`b_v1_diag_v4.md` H1（第 27–50 行）、H4（第 100–115 行）

**问题**：

审查报告同时包含以下两条判断：

- **H1**（第 34 行）：路径 A（URL 直接加载）"无需 `ImageService.getImage()`，无需 `ArrayBuffer`，无需 `PixelMap` 解码"，且标注为"首要行动：明确 `image_path` 语义"
- **H4**（第 105 行）："`api.ets` 层仍需新增 `requestRaw()` 方法，作为二进制数据获取的基础设施"；且将 `NetworkResult.rawBody` 不能承载二进制响应定性为"**高严重度—架构遗漏**"

这里存在未处理的逻辑矛盾：

1. **H1 的路径 A 若被确认，H4 的核心使用场景消失**：H1 的核心建议是优先确认 `image_path` 是否为 URL。若确认是 URL，则 `GET /image/{image_id}` 的二进制下载场景不再是病虫害记录图片展示的主路径，`requestRaw()` 仅在图片上传（`POST /image/upload`）中用于构建 multipart 请求体（而非响应体解析），其架构紧急度应显著降低
2. **审查报告未将 H1 结论的不确定性映射到 H4 的严重度评估中**：两条问题各自独立定性，未阐明"H1 的验证结果如何影响 H4 的严重度"。一个高严重度问题的判定依赖于另一个尚未得到确定答案的前提
3. **H4 的改进建议（拆分 `TextResult` / `BinaryResult`）与 H1 路径 A 的前提不兼容**：如果路径 A 被采纳，`NetworkResult` 在未来相当长时间内可能根本不需要 `ArrayBuffer` 分支

**判定**：两条高严重度问题之间存在未处理的依赖关系和潜在结论冲突，影响审查报告的逻辑自洽性。

## CH4 — 覆盖完备性：离线/弱网场景完全未覆盖

**位置**：`b_v1_diag_v4.md` 全文

**问题**：

审查视角为"可落地性"，且应用场景为**农业物联网**（摄像头、传感器部署在大田/大棚环境中），网络连接稳定性低于城市室内环境。但审查报告 **13 项问题中没有任何一项涉及离线/弱网场景的可落地性缺口**：

- 设计文档 `a_v1_design_v3.md` 的错误处理策略定义了各类错误的处理方式，但**缺失请求重试机制的定义**（例如：网络超时后是否自动重试？重试次数和退避策略是什么？）
- 设计文档 **未定义离线状态下的 UI 表现**（例如：数据卡顿/空值时是否展示"数据加载中"占位 vs "设备离线"提示？页面标题处是否展示连接状态指示器？）
- 设计文档 **未定义本地缓存策略**（IndexPage 首页在无网络连接时是否至少展示最后一次成功获取的快照？还是空白的 SensorCard？）
- **审查报告未发现这些缺口**，而是将其作为设计中被接受的部分

对于农业 IoT 场景而言，弱网韧性是"可落地性"的重要维度。审查报告将其完全遗漏，属于审查维度的严重覆盖不足。

**判定**：审查报告未覆盖 IoT 场景下弱网适应性的可落地性评估。

## CH5 — 逻辑完整性：严重度数量统计错误

**位置**：`b_v1_diag_v4.md` 第 21 行、第 344 行

**问题**：

审查报告自身存在数量统计的前后矛盾：

| 位置 | 声称 | 实际 |
|------|------|------|
| 第 21 行 | "v4 审查：**9 项**高严重度问题、4 项中严重度问题、3 项低严重度问题" | H1+H2+H3+H4+H5+H6+H8 = **7 项**HIGH（H7 已降级为中） |
| 第 344 行 | "当前 **9 项**高严重度问题" | 紧随其后的表格仅列出 H1、H2、H3、H4、H5、H6、H8 共 **7 项** |

即使将已降级为中的 H7 追溯计入，原始 HIGH 也仅有 8 项（H1–H8），仍然不等于 9。无论按何种口径计算，**9 项均无法成立**。

虽然这是格式层面的数值错误，但对于一份旨在评估质询质量的审查报告，"自身统计数据不准确"损害了报告的严谨性可信度。

**判定**：审查报告存在可验证的数量统计错误。
