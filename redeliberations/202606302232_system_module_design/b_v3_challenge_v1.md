## 四、新问题检测核验

诊断报告 §4 声明"本轮审查未发现新的报告级质量问题"，列有 2 项低于报告阈值的观察项。

### 4.1 观察项合理性评估

**观察项 1 — 流 C 未标注异步命令应答更新**：
- 经核实源文档 Flow C（行 549-564），流 C 在第 561 行显示 `◀──CMD Response────────│`（嵌入式 MCU → IoTDA），但 IoTDA 后续通过 `rule_cmd_response_forward` Webhook 转发至 Python API 的步骤未显式标注。
- 诊断报告将此列为"低于报告阈值"的理由是：该机制为全局机制（§4.2.3 对任意来源的命令应答均生效），且流 C 聚焦客户端交互路径。
- **评估**: 判断合理。Flow C 的主要目的是展示"客户端 → API → IoTDA → 设备"的同步调用链路，异步 Webhook 回调是独立的全局机制，在图内重复标注反而不利于清晰性。

**观察项 2 — 控制命令与决策动作的命名一致性**：
- 决策规则矩阵（行 289-302）中自动动作为 `spray ON`，但未见对应 `spray OFF` 的触发条件。
- 诊断报告的解读是"关闭操作可能依赖定时器或手动操作，此为系统行为设计选择，非严格缺陷"。
- **评估**: 判断合理。决策矩阵规则 #3/#6/#9/#12 触发 `spray ON` 后，喷淋何时停止是一个值得注意的系统行为决策，但确实不属于模块设计层面的缺陷——可以通过后续的定时器策略或手动控制来解决。诊断报告将其归入"低于阈值观察项"而非编号问题，严重度判断恰当。

### 4.2 潜在遗漏检查

以下为质询者对设计文档独立审阅后识别出的可能被诊断报告忽略的点：

1. **Webhook 端点的认证缺失**：§4.2 的三个 IoTDA Webhook 端点均标记为"不对客户端暴露"，但未讨论如何防止非 IoTDA 来源的恶意请求。§4.1 通用约定中认证方式为"初版可暂用无认证（VPS IP 白名单 + 内网）"。

   **评估**: 此为极微小的安全考量，v1.0 阶段 IP 白名单方案可接受。诊断报告不标记此项为预期合理——不构成实质性遗漏。

2. **图片上传端点缺少文件大小校验逻辑说明**：§4.7.1 请求体字段表中标注"最大 10MB"，但处理逻辑未显式说明大小校验步骤。

   **评估**: 此为极微小的实现细节（不超过 LOW 等级），FastAPI 的 `UploadFile` 天然支持通过配置限制上传大小，后端框架层面已有机制。诊断报告不标记此项——合理。

3. **docker-compose 端口绑定安全性**：§5.2 docker-compose 中 `ports: "8000:8000"` 将端口绑定到 `0.0.0.0`（所有接口），虽然这是公网 VPS 的必要配置，但未讨论 TLS/HTTPS 终止方案。

   **评估**: 诊断报告在 §3.1 系统总体设计交叉验证中未触及此点。但 v2 诊断报告 §3.5 曾标注 HTTP/HTTPS 一致性提醒，v3 诊断未重复提出属于合理的避免冗余。此点不构成 v3 诊断的覆盖缺失。

**结论**: 上述 3 项均属极微小边缘项，不构成诊断报告的实质性覆盖缺失。诊断报告的新问题检测工作充分。
