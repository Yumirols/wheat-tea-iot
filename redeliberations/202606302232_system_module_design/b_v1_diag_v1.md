## 二、严重问题（CRITICAL）

### [C1] 病虫害严重度等级数量与规格书矛盾

- **位置**: 第 2.2 节"严重度分级标准"表（行 160-165）
- **问题**: 设计文档定义了 4 级严重度（Normal/Mild/Moderate/Severe，编码 0-3），但 `system_specification.md` 第 4.2 节仅定义 3 级（Mild/Moderate/Severe），不存在"Normal（健康，未检出）"等级。
- **证据**:
  - 规格书原文: "*轻度 (Mild) → … *中度 (Moderate) → … *重度 (Severe) → …"
  - 设计文档新增了"0 - Normal — 健康，未检出"
- **影响**: 如果 AI 模型按 3 级输出而数据库/API 按 4 级设计，将导致严重度数据无法匹配，决策引擎逻辑直接出错。`severity_code` 取值范围也会产生离线偏差。
- **建议**: 与需求方确认。若确实需要"Normal"等级，必须同步修改 `system_specification.md` 并确保 AI 模型也输出此等级；否则设计文档应删除 0 级并改为 3 级（编码 1-3 或 0-2）。

### [C2] alarm_flag 位掩码完全未定义

- **位置**: 第 2.1 节"上报数据字段"表（行 119-135）及整篇设计文档
- **问题**: `alarm_flag` 字段被列为上报字段，数据库表也包含此字段，但设计文档中**没有任何地方定义其位掩码含义**。`DATA_INVENTORY.md` 第 2.12 节和四节已给出完整定义：
  - 0x01 高温, 0x02 低温, 0x04 高湿, 0x08 低湿, 0x10 低光照, 0x20 高 CO2, 0x40 低氮, 0x80 低磷
- **影响**: 任何阅读此设计文档的开发者（嵌入式端、后台、前端）都无法理解或处理 `alarm_flag` 的值。这是一个实现阻断性问题。
- **建议**: 从 `DATA_INVENTORY.md` 中直接引用 alarm_flag 位掩码表，作为传感器模块设计的一部分写入第 2.1 节。

### [C3] 告警阈值完全未定义——决策引擎无法实现

- **位置**: 第 2.4 节"决策引擎"描述（行 227）及整篇文档
- **问题**: Python API 决策引擎声称基于"阈值规则"生成自动控制决策，但设计文档中没有定义任何阈值。`DATA_INVENTORY.md` 已详细给出了每个传感器的告警阈值:
  - temperature > 38.0℃, < -10.0℃
  - humidity > 90.0%, < 15.0%
  - light < 100
  - co2 > 2000 ppm
  - soil_n < 30.0
  - soil_p < 10.0
- **影响**: 决策引擎的核心参数缺失，后端开发者需要到另一个文档查找方可实现，且设计文档作为"设计真相源"的完整性受损。
- **建议**: 在决策引擎相关章节（建议第 2.4 节或新增专节）中明确列出所有传感器的告警阈值及对应的自动控制动作。

### [C4] 土壤 NPK 推导公式缺失

- **位置**: 第 2.1 节"上报数据字段"表（行 119-135）及第 6.5 节
- **问题**: 设计文档将 soil_n/soil_p/soil_k 标记为"推导/模拟"，第 6.5 节再次说明为推导值，但**未给出具体推导公式**。`DATA_INVENTORY.md` 第 2.5-2.7 节明确定义了公式：
  - `soil_n = 45.0 + temperature * 0.2`
  - `soil_p = 18.0 + humidity * 0.1`
  - `soil_k = 50.0 + light * 0.02`
- **影响**: 嵌入式端开发者无法实现 NPK 推导计算，不同实现者可能得出不一致的结果。
- **建议**: 在第 2.1 节或第 6.5 节中明确写入三个推导公式。

## 三、高优先级问题（HIGH）

### [H1] 病虫害记录 API 响应中缺少 image_path 字段

- **位置**: 第 4.4.1 节 GET `/api/v1/disease/records` 成功响应（行 713-736）
- **问题**: 数据库表 `disease_records` 包含 `image_path VARCHAR(512)` 字段，但查询 API 返回的 record 对象中不包含 `image_path`。这意味着存储的图片路径永远无法被客户端获取，图片功能形同虚设。
- **建议**: 在 `/api/v1/disease/records` 和 `/api/v1/disease/stats` 的响应 record 对象中增加 `image_path` 字段。

### [H2] 缺少图片上传/获取 API 端点

- **位置**: 第 4.7 节接口清单
- **问题**: 系统涉及摄像头图像采集、AI 识别、病虫害记录的 `image_path` 存储，但 10 个 API 端点中没有任何一个支持图片上传或获取。当前设计无法将边缘端采集的图像传输到服务器保存。
- **建议**: 增加至少两个端点：
  - `POST /api/v1/image/upload` — 边缘端或上位机上传病虫害图像
  - `GET /api/v1/image/{image_id}` — 客户端获取图像文件

### [H3] 缺少需求规格书明确要求的热力图功能

- **位置**: 全文
- **问题**: `system_specification.md` 第 2.3 节明确要求"生成病虫害防治建议及**病虫害分布热力图**"。设计文档的防治建议接口（4.6 节）仅返回文本建议，没有任何热力图数据接口或相关功能描述。
- **建议**: 增加热力图数据接口（如 `GET /api/v1/disease/heatmap`），或在现有防治建议接口 `data` 中增加热力图数据结构。

### [H4] 缺少环境数据与病虫害发生的联动监测分析

- **位置**: 全文
- **问题**: `system_specification.md` 第 2.1 节要求"实现环境数据与病虫害发生概率的联动监测"。设计文档将传感器数据和病虫害数据作为两条独立的数据流处理，没有定义任何跨域关联分析功能。
- **建议**: 在决策引擎或分析模块中增加联动监测逻辑，明确当特定环境条件（如高温高湿）与病虫害检测同时出现时的处理规则，并提供相应的查询/统计 API。

### [H5] 设备在线/离线判定机制未定义

- **位置**: 第 4.3.3 节 GET `/api/v1/device/list` 响应（行 670-686）
- **问题**: 响应中包含 `online: true` 和 `last_seen` 字段，但设计文档没有定义如何判定设备在线/离线。需要明确：超时阈值是多少（如 30s 内无数据上报即判定离线），以及这个阈值是否有依据。
- **建议**: 在第 2.4 节或设备管理相关章节中明确在线判定逻辑，建议阈值 30s（3 个上报周期）。

### [H6] 鸿蒙端消息推送机制空白

- **位置**: 第 2.6 节"鸿蒙应用"职责（行 319-328）
- **问题**: 设计文档声明鸿蒙 App 支持"病虫害预警消息推送（接收重度检测通知）"，但未说明推送采用何种技术机制（HTTP 轮询？WebSocket？华为 Push Kit？）。技术选型直接决定了后台是否需要维护长连接、是否需要集成华为推送服务。
- **建议**: 明确选择推送方案并记录设计决策。若采用轮询，说明轮询周期；若采用 WebSocket，增加 WebSocket 端点定义；若采用华为 Push Kit，增加集成说明。

## 四、中优先级问题（MEDIUM）

### [M1] IoTDA 双 Webhook 端点路由机制未说明

- **位置**: 第 4.2.1 与 4.2.2 节
- **问题**: 设计文档为传感器数据和 AI 识别结果定义了**两个独立的 Webhook 接收端点**（`/api/v1/iotda/properties/report` 和 `/api/v1/iotda/ai/report`）。但华为云 IoTDA 的数据流转规则是基于 service_id 或 resource 字段进行路由配置的——设计文档没有说明如何配置 IoTDA 规则使两类数据分别路由到不同端点，也没有说明是否两个端点实际由同一处理函数接收后在代码内部分发。
- **建议**: 明确 IoTDA 规则配置方案，或说明采用单端点接收后在代码内按 `service_id` 分发（此时两个端点的设计应合并）。

### [M2] 数据库表命名与参考文档不一致且未经调和

- **位置**: 第 2.5 节表名（行 243, 270, 294）
- **问题**: `system_architecture_relationship.md` 第 2.5 节将环境数据表称为 `sensor_data`，而 `DATA_INVENTORY.md` 和设计文档均使用 `sensor_snapshot`。设计文档未对此差异做任何说明。
- **建议**: 在设计文档中明确选择 `sensor_snapshot` 并注明与架构关系说明文档的差异，或协调统一两者。

### [M3] 决策引擎自动控制规则仅有一条示例

- **位置**: 第 3.2 节流 B 注释（行 389-391）
- **问题**: 数据流 B 中决策引擎的注释仅给出了一条规则："如 Severity=Severe 则自动触发喷淋"。但实际场景远不止此——需要根据不同的病虫害类型、严重度、环境条件组合生成不同控制决策。仅有这一条例无法支撑决策引擎的完整实现。
- **建议**: 补充完整的决策规则矩阵，至少覆盖：各类病虫害 × 各严重度 × 关键环境条件 的组合及对应控制动作。

### [M4] 缺少数据保留/清理策略

- **位置**: 全文
- **问题**: 传感器数据按 10s 周期上报，每天产生 8,640 条记录，长期运行将迅速膨胀。设计文档没有提及任何数据保留策略（如只保留最近 N 天、定期归档、按天聚合等）。
- **建议**: 增加数据生命周期管理说明，至少给出建议的数据保留周期。

### [M5] OLED I2C 地址缺失

- **位置**: 第 2.1 节 GPIO 表（行 100-118）
- **问题**: `DATA_INVENTORY.md` 第 4 节明确标注 OLED SSD1306 的 I2C 地址为 `0x3C`，设计文档的 GPIO 表未包含此信息。
- **建议**: 在 GPIO 表的 OLED 行增加 I2C 地址标注。

### [M6] 超声波传感器异常值处理未说明

- **位置**: 第 2.1 节"上报数据字段"表（行 119-135）
- **问题**: `DATA_INVENTORY.md` 第 2.8 节注明 distance 字段在 HC-SR04 超时情况下返回 -1，设计文档未说明此异常值的含义及处理方式。
- **建议**: 在数据字段说明中注明 distance 的特殊值含义（-1=超时/无目标），并在 API 后台和数据库层面说明如何处理 -1 值。
