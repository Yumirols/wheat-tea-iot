## 三、5 项新发现问题核验

### N1 — 图片存储持久化机制未定义（MEDIUM）

- **诊断位置**: §4.7.1 + §5.2 docker-compose
- **诊断声明**: docker-compose.yml 中未见图片存储相关 volume 挂载；图片存储在容器内部则容器重建后丢失
- **设计文档验证**: §5.2 docker-compose（行 1432-1433）仅定义 `db_data` volume，无图片存储 volume；§4.7.1 返回 `image_path` 但未说明物理存储位置
- **核验结果**: **成立**。诊断正确识别了图片持久化机制的缺失
- **严重度合理性**: MEDIUM 定级合理 — 图片存取是 H2 需求修复后的支撑性功能，volume 缺失将导致 v2 修复的图片功能在生产环境中不可用

### N2 — IoTDA 命令应答 Webhook 回传路径不完整（MEDIUM）

- **诊断位置**: §3.2 流 B + §4.2 Webhook 配置
- **诊断声明**: 流 B 显示命令应答通过 Webhook 回传，但 §4.2 仅定义 sensor_forward 和 ai_forward 两条规则，缺少命令应答转发规则
- **设计文档验证**: §4.2 行 627-630 仅定义两条规则；§3.2 流 B 行 540-543 显示命令应答通过 "MQTT PUB → IoTDA → HTTP POST (Webhook)" 路径回传，但标注 "or 状态同步" 表明诊断也承认存在替代方案
- **核验结果**: **成立**。诊断正确识别了命令应答回传闭环的不完整性
- **严重度合理性**: MEDIUM 定级合理 — 命令应答路径不完整影响 control_logs 中 result_code 的正确更新，属功能性缺口但与核心数据流独立
- **附注**: 诊断建议给出了方案 A（新增 Webhook 规则）和方案 B（轮询 IoTDA 状态），二选一建议具有建设性

### N3 — 决策规则矩阵边界值未明确开闭区间（LOW）

- **诊断位置**: §2.4 决策规则矩阵
- **诊断声明**: 矩阵中 "15-25℃"、"50-80%" 等范围未明确开闭性
- **设计文档验证**: §2.4 行 291-300 共 7 处使用范围写法（如 `temperature 15-25℃`、`humidity 50-80%`、`temperature 20-30℃`），确未标注 ≤ 或 <
- **核验结果**: **成立**
- **严重度合理性**: LOW 定级合理 — 边界 off-by-one 偏差仅在高概率边界值出现时才产生影响，且实现阶段可协商

### N4 — 数据聚合后存储的表结构未定义（LOW）

- **诊断位置**: §2.5 数据保留与清理策略
- **诊断声明**: 策略声明 30 天前数据按天聚合后仅保留聚合记录，但未定义聚合表 DDL
- **设计文档验证**: §2.5 行 415 策略声明 "按天聚合（取均值/最大/最小）后仅保留聚合记录"，但 DDL 仅含 sensor_snapshot/disease_records/control_logs 三张表，缺少聚合表定义
- **核验结果**: **成立**
- **严重度合理性**: LOW 定级合理 — 聚合表结构可在详细设计阶段定义，不影响当前阶段的架构决策

### N5 — Python API 端 Webhook 异常处理策略未说明（LOW）

- **诊断位置**: §4.2.1 和 §4.2.2 Webhook 端点
- **诊断声明**: Webhook 端点仅定义 HTTP 200 成功响应，未说明 DB 写入失败时应返回何种状态码
- **设计文档验证**: §4.2.1 处理逻辑步骤 5 为 "返回 200 OK"；§4.2.2 处理逻辑步骤 4 为 "返回 200 OK"，均无错误状态码说明
- **核验结果**: **成立**
- **严重度合理性**: LOW 定级合理 — 错误处理策略属于实现细节，但若忽略可能导致 IoTDA 静默丢数据

**5 项新发现问题全部经原设计文档验证，均成立且严重度定级合理。**
