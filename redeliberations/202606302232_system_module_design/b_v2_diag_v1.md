## 四、新发现问题

除原 30 项问题外，本轮审查新发现以下 5 项问题：

### [N1] MEDIUM — 图片存储持久化机制未定义

- **位置**: §4.7.1 图片上传端点 + §5.2 docker-compose
- **问题**: 图片上传端点定义了上传成功返回 `image_path`，但未说明图片在 VPS 上的物理存储位置（本地文件系统路径、Docker 挂载卷、还是外部对象存储）。docker-compose.yml 中未见图片存储相关的 volume 挂载。若图片存储在容器内部，容器重启或重建后图片将丢失。
- **影响**: Docker 容器重建后历史图片不可访问，GET /api/v1/image/{image_id} 将返回 1002 "image not found"。
- **建议**: 在 docker-compose 中增加图片存储 volume 挂载（如 `- ./images:/app/images`），或从 §4.7 端点说明中补充存储策略（本地磁盘 / 外部对象存储）。图片路径格式应为相对于 volume 挂载点的路径。

### [N2] MEDIUM — IoTDA 命令应答回传路径不完整

- **位置**: §3.2 流 B 时序图 + §4.2 Webhook 配置
- **问题**: 流 B 的自动控制流程中，命令应答通过 "IoTDA → HTTP POST (Webhook)" 回传 Python API，但 §4.2 仅定义了 `rule_sensor_forward`（service_id=farmeye_env）和 `rule_ai_forward`（service_id=farmeye_ai）两条 Webhook 规则，未定义命令应答的 Webhook 路由。命令应答的 MQTT Topic 为 `.../sys/commands/response/...`，若 IoTDA 未配置对应的数据转发规则，应答将无法推送到 Python API。
- **影响**: Python API 无法感知命令是否被设备实际执行（仅能通过 IoTDA API 返回获知"命令已下发"，而非"命令已执行"）。control_logs 的 result_code 字段将无法被正确更新。
- **建议**: 明确命令应答的处理方案（二选一）：
  - 方案 A：在 IoTDA 增加第三条转发规则 `rule_cmd_response_forward`，将命令应答通过 Webhook 转发至 Python API
  - 方案 B：Python API 通过轮询 IoTDA 设备影子/状态接口获取命令执行结果
  - 推荐方案 A，并在 §3.2 流 B 时序图中明确标注命令应答的回传路径

### [N3] LOW — 决策规则矩阵中环境条件边界值未明确开闭区间

- **位置**: §2.4 决策规则矩阵
- **问题**: 矩阵中多处使用范围条件如 "temperature 15-25℃"、"humidity 50-80%"，但未明确区间的开闭性（15℃ 是否触发？25℃ 是否触发？）。在实际实现时，若前后端对边界值处理不一致，可能导致同一条件下决策结果不同。
- **影响**: 实现时可能产生边界条件的 off-by-one 偏差，但属于可协商的实现细节。
- **建议**: 将范围表示为 `15℃ ≤ temperature ≤ 25℃` 或 `15℃ < temperature < 25℃` 的明确形式。

### [N4] LOW — 数据聚合后存储的表结构未定义

- **位置**: §2.5 数据保留与清理策略
- **问题**: sensor_snapshot 策略声明"30 天前数据按天聚合（取均值/最大/最小）后仅保留聚合记录"，但未定义聚合表的结构（如 `sensor_daily_aggregation` 表）。定时任务需要写入聚合数据的表不存在于数据表规划中。
- **影响**: 定时任务的实现者在编码时需自行设计聚合表结构，可能导致与原始表字段不一致。
- **建议**: 补充聚合表 DDL 或在策略说明中注明"聚合表结构将在详细设计阶段定义"。

### [N5] LOW — Python API 端 Webhook 异常处理策略未说明

- **位置**: §4.2.1 和 §4.2.2 Webhook 端点
- **问题**: 两个 IoTDA Webhook 端点均只定义了成功响应（HTTP 200），未说明数据库写入失败时应返回何种 HTTP 状态码。IoTDA 数据流转规则有重试机制（默认重试 3 次，间隔 30s，已在 §4.2 配置要点中说明），若 API 在 DB 写入失败时仍返回 200，IoTDA 将认为转发成功而不重试，导致数据丢失。
- **影响**: 数据库临时不可用时，IoTDA 转发的数据会被静默丢弃，无法通过重试机制恢复。
- **建议**: 在 §4.2.1 和 §4.2.2 的"处理逻辑"步骤中增加错误处理说明：数据库写入失败时返回 HTTP 500（触发 IoTDA 重试），并记录错误日志。
