## 第二项质询：发现2的评价标准超出审查声明的范围边界

### 类型

覆盖完备性 / 逻辑完整性 — 审查维度偏离了审查报告的自身声明范围

### 位置

- `b_v4_diag_v2.md` 第三节"审查概述"第7行："审查范围：组件A问题诊断报告（diagnosis模式）的可操作性评估"
- 发现2（`online` 字段 ORM-DDL 与 SQL-DDL 默认值语义不一致）

### 范围偏离分析

本审查声明的审查范围是"诊断报告的可操作性评估"。发现2指出诊断报告未覆盖 `online` 字段的 ORM-DDL 与 SQL-DDL 默认值不一致：

> `control.py:44` 的 ORM 定义 `online = Column(Boolean, default=False)` 仅含 Python 侧默认值，不含 `server_default`。这意味着通过 `Base.metadata.create_all()` 生成的 DDL 为 `online BOOLEAN`（无 DEFAULT 子句），与 `01_create_tables.sql` 中的 `online BOOLEAN DEFAULT FALSE` 不一致。

然而发现2**本身**确认了两项关键事实：

1. **"该不一致不引起 ERROR（PostgreSQL 允许 NULL）"** — 即该"不一致"不是两类测试失败的成因，与诊断报告的核心任务无关。
2. **"对可操作性的影响：无"** — 即该发现对本审查报告自身声明的"可操作性评估"主题无贡献。

一份声称以"可操作性评估"为范围的审查报告，如果其发现的自评影响为"无"且与两大核心问题无关，则该发现不应被列为负面发现，更不应被用作评分扣分依据。

此外，该发现的具体内容 —— `online = Column(Boolean, default=False)` 缺少 `server_default` —— 是 SQLAlchemy 中**标准且有意的 Python 侧默认值设计模式**，而非代码缺陷。当所有 INSERT 操作均通过 ORM 执行时，Python 侧 `default` 提供正确的默认值，无需数据库侧的 `server_default`。将该模式标注为"不一致"暗示其为待修复事项，与标准实践不符。

### 与评分扣分的关联

第三节对"边界情况和安全隐患"评 3/5 的扣分依据之二为"未提 online 字段 ORM-DDL default 不一致"。既然发现2对可操作性的自评影响为"无"，则该扣分不构成对可操作性评估的贡献 —— 它扣的是诊断报告在一条不相关的维度上的分数。由此，第三节中"边界情况和安全隐患"的评分需要重新审视。
